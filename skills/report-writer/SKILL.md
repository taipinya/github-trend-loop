# report-writer

## Purpose

根据 Top 3 GitHub repo 的结构化数据生成中文 Markdown 报告。报告面向计算机专业大学生，重点帮助读者理解业界生产实践、前沿技术方向和可借鉴的项目灵感。

## Language

- 报告必须使用中文。
- 表达要具体、有信息价值，避免只写泛泛的推荐语。
- 如果本次是 baseline，必须明确说明下一次运行才有 3 天增量数据。
- 不要把报告写成安装教程或操作手册。

## Required Sections Per Project

每个项目必须包含：

- URL
- 项目简介：说明项目属于什么领域、可能使用什么技术原理、完成什么任务。
- 为什么值得关注
- README 摘要：添加中文说明，解释 README 的结构和内容线索。
- 具体怎么用：面向学习和理解项目，不写成安装教程。
- 适合谁关注
- 应用场景示例：至少 3 条，贴合具体工业生产或真实业务流程。
- 风险或局限
- 数据限制：说明 GitHub 没有直接提供全站 3 天增长榜、增量来自本地 snapshot、Trending 来自 HTML 抓取。

## Writing Guidance

- 使用 README 摘要、description、topics、language、stars、forks、subscribers、push 时间和评分结果。
- 不能假装已经实际安装或运行项目，除非有明确的人工作业记录。
- 不生成“一句话判断”“具体使用方式”“下一步验证建议”章节。
- 必须使用“具体怎么用”和“适合谁关注”作为小节标题。
- 不写“可用于模型实验、推理链路验证、AI 应用原型或研究复现”这类空泛句子。
- 应用场景应尽量具体，例如客服质检、企业知识库、代码审查、研发平台、工厂巡检、机器人仓储、设备异常检测等。
- 风险说明必须覆盖维护活跃度、许可证、README 完整度、issues 或生产可用性等至少一个方面。

## Human Gate

报告生成后只能作为草稿。是否分享、发布或采用，需要人工审核。
